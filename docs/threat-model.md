**Status:** Draft 1

## Scope

This document enumerates the security-relevant assets, threats, and mitigations in scope for Tessera v0.1 through v1.0. It is the authoritative source for security decisions. Features and code must be checked against this model; when a feature cannot be delivered without violating a mitigation, the threat model is re-opened, not the feature.

## Assets

| Asset                            | Description                                                                     | Sensitivity                                                                |
| -------------------------------- | ------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| A1 — Vault file                  | `~/.tessera/vault.db`: facets, capabilities, audit log, vector indexes          | High. Contains the user's full portable context across identity, preferences, workflows, projects, and style. |
| A2 — Capability tokens (live)    | Raw tokens presented by MCP-connected AI tools in requests                      | High. Bearer; possession = tool impersonation across the token's granted facet-type scopes. |
| A3 — Capability tokens (at rest) | Hashed tokens in `capabilities.token_hash`                                      | Low on their own; high when correlated with A2.                            |
| A4 — Daemon process              | `tesserad` running, holding the vault open, serving MCP                         | Medium. Compromise = persistent access to A1, A2.                          |
| A5 — Config files                | `~/.tessera/config.yaml`, client-side MCP configs (Claude Desktop, Codex, etc.) | Medium. Client configs typically hold the raw token.                       |
| A6 — Model adapter credentials   | OpenAI / Voyage / Cohere API keys, when cloud adapters are opt-in               | High when present. Not present in all-local mode.                          |
| A7 — Audit log                   | Append-only record of operations                                                | Medium. Reveals activity patterns; compromise = deniability loss.          |
| A8 — Embeddings                  | Dense vectors of facet content                                                  | Low on their own; high via embedding inversion attacks at scale.           |

## Actors

| Actor                                   | Description                                                                      | Default trust                                           |
| --------------------------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------- |
| T1 — User                               | The human on the machine                                                         | Trusted                                                 |
| T2 — Connected AI tool                  | Any MCP client (Claude Desktop, Claude Code, Cursor, Codex, ChatGPT Dev Mode, autonomous agent) running with a valid capability token | Trusted within the scopes granted to its token          |
| T3 — Co-located process                 | Any other process running as the same OS user                                    | **Untrusted**                                           |
| T4 — Co-located user (shared host)      | Another OS user on the same machine (SSH, multi-user dev box)                    | Untrusted                                               |
| T5 — Remote network attacker            | Attacker on the LAN or internet                                                  | Untrusted; must not be reachable by default             |
| T6 — Physical attacker with disk access | Attacker who can read the filesystem offline (stolen laptop, unencrypted backup) | Untrusted                                               |
| T7 — Cloud model provider               | OpenAI / Voyage / Cohere when opt-in adapters are used                           | Semi-trusted (they see queries + embeddings, not vault) |
| T8 — Sync backend (v0.5+)               | BYO S3-compatible storage                                                        | Untrusted (must only see ciphertext)                    |

## STRIDE by surface

### S1 — Vault file (A1)

| Threat                                             | Category               | Vector                                                                      | Mitigation                                                                                                                                                 | Status                               |
| -------------------------------------------------- | ---------------------- | --------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| Offline read of vault by attacker with disk access | Information disclosure | T6 reads `vault.db` from stolen disk, unencrypted backup, emailed copy      | **Encryption at rest**: sqlcipher or libsodium-encrypted pages, key derived via argon2id from user passphrase. See `system-design.md §Encryption at rest`. | v0.1 mandatory                       |
| Offline modification of vault                      | Tampering              | T6 edits `vault.db` to plant facets, rewrite audit log, insert capabilities | Encryption at rest makes modification require the key; HMAC-chained audit log makes tampering detectable.                                                  | v0.1 (encryption), v0.3 (HMAC chain) |
| Accidental exposure via user action                | Information disclosure | User emails vault, uploads to Dropbox unencrypted, commits to git           | Default encryption means the exported file is ciphertext. Docs explicitly warn against sharing the passphrase.                                             | v0.1                                 |
| Cross-tool scope leak                              | Information disclosure | A token granted `read: [preference, workflow]` to ChatGPT returns `style` facets to that tool | Per-facet-type scope check is the first gate of every `recall`/`show`/`list_facets`/`stats` path (see `system-design.md §Trust & capability tokens`). Integration tests cover every (scope-set, facet-type) combination. v1.0 multi-user isolation additionally enforces `user_id` foreign key on every SELECT. | v0.1 (scope check); v1.0 (multi-user)|

### S2.1 — Agent profile linkage (V0.5-P2, ADR 0017)

| Threat                                              | Category               | Vector                                                                                                                                                                                                  | Mitigation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  | Status                  |
| --------------------------------------------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| Token grants `read:agent_profile` to wrong tool     | Information disclosure | A capability scoped to read agent profiles is issued to a tool that should not see another agent's `purpose` / `inputs` / `cadence` (e.g., a third-party connector vs. the user's own orchestrator)     | `read:agent_profile` is a per-scope grant, not implicit. The MCP `get_agent_profile` tool runs the standard scope check, then performs an additional `agent_id` guard on the returned row — a token whose `verified.agent_id` does not match the profile facet's `agent_id` receives `null` regardless of whether the ULID was leaked. Cross-agent reads cannot succeed even with a valid scope. Integration test: `test_get_agent_profile_returns_none_for_other_agent`.                                                  | v0.5-P2                 |
| Auth pipeline reads profile metadata for decisions  | Elevation of privilege | A regression has the auth/token-verify path consult `agent_profile.metadata` (e.g., `verification_ref`) when issuing or rotating a capability                                                          | Boundary statement (ADR 0017): `agents` is the JWT subject store; `agent_profile` is recallable context. Auth code (`src/tessera/auth/`) reads only `agents`. The deferrable FK column on `agents.profile_facet_external_id` is structural — token issuance and verification never join through it. A failing test of the form "verifying a token does not load the profile facet" guards the boundary going forward.                                                                                                       | v0.5-P2                 |
| Stale link to a soft-deleted profile                | Integrity              | A profile facet is soft-deleted (or hard-deleted via `vault vacuum`) but `agents.profile_facet_external_id` still references its `external_id`                                                          | `get_active_for_agent` returns `None` when the linked facet is missing or soft-deleted; callers can repair by re-registering or by calling `clear_active_link` (which writes an `agent_profile_link_cleared` audit row). The deferrable FK does not auto-NULL the column — that is intentional so forensics can reconstruct how the link drifted. The MCP boundary surfaces the `None` result honestly rather than fabricating an empty profile.                                                                            | v0.5-P2                 |
| Profile content leaks via audit payload             | Information disclosure | The `agent_profile_link_set` / `agent_profile_link_cleared` audit ops carry profile narrative or structured metadata into the audit log                                                                | Audit allowlist is closed at the source (`src/tessera/vault/audit.py`): `agent_profile_link_set` permits only `prior_external_id`; `agent_profile_link_cleared` permits the empty set. The facet itself rides the existing `facet_inserted` op which carries `facet_type` + content-hash prefix only. Profile content and structured metadata never enter audit rows (§S4 unchanged).                                                                                                                                       | v0.5-P2                 |

### S2.2 — Verification checklist + retrospective linkage (V0.5-P3, ADR 0018)

| Threat                                                          | Category               | Vector                                                                                                                                                                                                                                                  | Mitigation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          | Status                  |
| --------------------------------------------------------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| Cross-agent `agent_ref` poisoning                               | Tampering              | A token scoped to write checklists / retrospectives could plant an artifact whose `agent_ref` points at another agent's profile. The row's own `agent_id` stays the writer's, so the storage layer accepts it; the cross-agent reference is the harm.   | The MCP boundary in `register_checklist` and `record_retrospective` runs `_enforce_same_agent_profile_ref` before delegating to the vault module. The guard fetches the referenced profile and rejects the call when `profile.agent_id != verified.agent_id` with `ValidationError("agent_ref ... belongs to a different agent")`. Integration test `test_register_checklist_blocks_cross_agent_ref` exercises the path.                                                                                                                                                            | v0.5-P3                 |
| Stale or dangling `verification_ref` on `agent_profile`         | Integrity              | The agent_profile's `verification_ref` points at a checklist that has since been soft-deleted (or hard-deleted via `vault vacuum`). Callers asking for the canonical checklist receive a misleading hit.                                                | `verification.get_canonical_for_profile` returns `None` when the linked row is missing or soft-deleted; the MCP `list_checks_for_agent` surfaces `null` honestly rather than fabricating a row. Callers re-register the profile (or update `verification_ref` to a live checklist) to repair. The deferrable FK on `agents.profile_facet_external_id` is unrelated; `verification_ref` lives in profile metadata and is intentionally not foreign-keyed at the schema level.                                                                                                       | v0.5-P3                 |
| SWCR retrospective augmentation leaks across agents             | Information disclosure | The retrieval pipeline augments SWCR with retrospectives whose `agent_ref` matches an `agent_profile` candidate. A bug in the augmenter could pull in retrospectives owned by another `agent_id`.                                                       | `retrospectives.recent_for_agent` filters by both `agent_id` (the calling token's principal) and `agent_ref` (the metadata-side profile pointer). The augmenter passes `ctx.agent_id` from the verified capability, not from the candidate row. Integration test `test_swcr_augments_with_recent_retrospectives` verifies the augmentation surface; the cross-agent leakage path is covered by the `recent_for_agent` agent_id filter and by the storage-layer agent-id guard on every facet read.                                                                                | v0.5-P3                 |
| Retrospective accumulation poisons audit / SWCR over time       | Availability / drift   | Retrospectives accumulate per-run with `volatility='persistent'` by default. Over time, an agent with thousands of retrospectives skews SWCR bundles and bloats the audit trail.                                                                        | Callers may revolatilize old retrospectives to `volatility='session'` (per ADR 0016) or `forget` them once their lessons are absorbed into the profile or checklist. The `retrieval.swcr.retrospective_window` parameter caps how many retrospectives augment SWCR per recall (default 3). Tests around bundle composition guard against the most recent retrospectives dominating; the documented escape hatch is the volatility model.                                                                                                                                          | v0.5-P3                 |
| Checklist `pass_criteria` mistaken for enforcement              | Authentication / trust | A caller treats `pass_criteria` as a programmatic gate the daemon enforces. Tessera stores the checklist; it never executes the checks. A malicious or buggy agent could ignore the checklist entirely.                                                 | ADR 0018 §Boundary statement is explicit: verification is a run-gate, not a guarantee. Tessera registers; callers execute. The MCP tool description says so; `system-design.md` repeats it; the boundary mirrors ADR 0020's stance on automations. No checklist execution engine ships in v0.5; that surface is bounded by the AgenticOS Layer 8 stance against in-process plugins.                                                                                                                                                                                                | v0.5-P3                 |

### S2.3 — Compiled artifact (Playbook) surface (V0.5-P4, ADR 0019)

| Threat                                                 | Category               | Vector                                                                                                                                                                | Mitigation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | Status      |
| ------------------------------------------------------ | ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------- |
| Cross-agent compiled-artifact disclosure               | Information disclosure | A token scoped to `read:compiled_notebook` for agent A is used to read agent B's playbook by guessing its ULID.                                                       | The MCP `get_compiled_artifact` tool runs the standard scope check, then performs an explicit `agent_id` guard on the returned row — a token whose `verified.agent_id` does not match the artifact's `agent_id` receives `null`. Cross-agent reads cannot succeed even with a valid scope. Integration test `test_get_compiled_artifact_blocks_cross_agent_read` exercises the path against a real two-agent vault.                                                                                                                                                                                                                                                                                                                                  | v0.5-P4     |
| Synthesized state without auditability                 | Repudiation            | A `compiled_notebook` facet contains content that did not exist in any source. A reviewer (or the user weeks later) cannot tell whether the row was tampered with.    | V0.5-P8 hard ship-gate: every `register_compiled_artifact` writes a `compiled_artifact_registered` row through the chain-aware `audit_log_append` insert path. The `test_chain_full_walk_clean_with_compiled_artifacts` security test asserts a vault that has ingested compiled artifacts walks the chain cleanly. ADR 0019 §Rationale (8) and ADR 0021 §Public language guardrails name the gate as the load-bearing trust posture for write-time mode reaching users. The chain detects accidental corruption + non-recomputed tampering on the artifact row + the audit row; recompute-capable tampering and pre-upgrade tampering remain out of scope by ADR 0021 §Security claim — exact boundary.                                                  | v0.5-P4 / P8 |
| In-process LLM via the daemon                          | Trust posture          | A future "convenience" `compile_now()` API embeds a model server in the daemon, reintroducing the runtime closure ADR 0014 just removed.                              | ADR 0019 §Boundary statement: Tessera stores; the caller compiles. The two-call API (`list_compile_sources` / `recall` to read, `register_compiled_artifact` to write) is the only write path. There is no `compile_now()` API, no in-process LLM, no auto-compile. Mirrors ADR 0018 (Tessera stores checklists; callers execute) and ADR 0020 (Tessera registers automations; callers run). A future surface that wants in-process compilation opens its own ADR before extending the boundary.                                                                                                                                                                                                                                                       | v0.5-P4     |
| Source-facet tampering before compile                  | Tampering              | A user (or a buggy upstream tool) edits a source facet's `metadata.compile_into` array between source enumeration and the `register_compiled_artifact` call.          | The compile is caller-side; Tessera stores whatever bundle the caller delivers. The `compiled_artifacts.source_facets` JSON column records the source ULIDs the caller claims to have read, and the chain insert binds them into the audit hash so source-list manipulation surfaces under `tessera audit verify`. The metadata edit on the source facet itself is covered by the source's own `facet_inserted` chain row (the row's content_hash + payload are part of its chain entry). A future ADR may tighten this by validating the source list against recently-recalled facets at the MCP layer.                                                                                                                                              | v0.5-P4     |
| Bypass write-path validation via raw `capture`         | Trust posture          | A write-scoped caller targets `facet_type='compiled_notebook'` through the generic `capture` tool, sidestepping the structured-source / compiler-version contract.    | The MCP `capture` tool already carries an explicit gate that rejects `facet_type='agent_profile'` for the same reason; the same pattern applies to `compiled_notebook` once a real misuse pattern shows up. v0.5-P4 ships without that gate because the `register_compiled_artifact` path is the documented public surface and the generic `capture` path leaves the row in a half-formed state (no paired `compiled_artifacts` row, no source provenance). Real-user signal will tell us whether the gate needs the same hard rejection. Tracked as a v0.5.x revisit trigger.                                                                                                                                                                          | v0.5-P4     |
| Cross-agent stale-flag cascade via leaked source ULID  | Tampering              | A leaked source-facet ULID surfaces in another agent's `compiled_artifacts.source_facets` JSON list (e.g., a forged write through an unrelated tool, a sync-import bug). A naïve stale-flagger walking by ULID alone would flip the second agent's artifacts when the first agent mutates its source. | `vault.compiled.mark_stale_for_source` filters by `agent_id` at the SELECT — every cascade is scoped to the mutating row's agent. The `_verify_sources_belong_to_agent` write-path gate already rejects cross-agent source lists before the artifact lands; the staleness primitive's own filter is the second line of defence so the invariant holds even when the write-path gate is breached. Security test `test_cross_agent_mutation_does_not_cascade` plants the impossible-via-public-write shape directly to prove the filter is independently load-bearing.                                                                                                                                                                                                       | v0.5-P6     |
| Synthesized stale-flag inflation by mutation flooding  | Repudiation            | A repeatedly-mutating source facet emits a fresh `compiled_artifact_marked_stale` audit row per cycle, padding the chain with synthesized cascade events.             | `mark_stale_for_source` is idempotent at the WHERE clause: artifacts already at `is_stale=1` are skipped, so a second mutation against the same source emits no second audit row for the same artifact. The chain only grows by one row per genuine 0→1 transition. Until the user (or compiler) recompiles the artifact and resets `is_stale=0`, no further cascade rows append regardless of how many source mutations follow. Security test `test_chain_full_walk_clean_with_compiled_staleness` walks a vault containing both a stale event and a follow-up no-op mutation, asserting exactly one cascade row. The chain itself remains tamper-evident under ADR 0021's claim boundary; the idempotency property keeps the chain length proportional to genuine state transitions, not user-visible mutation count. | v0.5-P6     |
| Stale-flag side-effect omitted on a new mutation path  | Tampering              | A future contributor adds a fourth source-mutation path (e.g., a hard-rewrite skill body) without wiring `mark_stale_for_source`. Playbooks citing the source go stale silently, leaving the compiler with stale narratives the user trusts.            | The wiring lives in three named call-sites (`vault.capture.capture`, `vault.facets.soft_delete`, `vault.skills.update_procedure`). `mark_stale_for_source` rejects unknown `source_op` values via an explicit allowlist (`facet_inserted` / `facet_soft_deleted` / `skill_procedure_updated`) so a new path that calls the helper without extending the allowlist fails fast at runtime instead of recording a malformed audit row. Security: tests assert each path emits the cascade for its declared `source_op`; a new path needs a new test row. Documented as the "narrow flagger contract" in ADR 0019 §Rationale (6) — direct membership only, no transitive walk; a future expansion ADR opens its own scope review. | v0.5-P6     |

### S2.4 — Automation registry surface (V0.5-P5, ADR 0020)

| Threat                                                  | Category              | Vector                                                                                                                                                                | Mitigation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                | Status      |
| ------------------------------------------------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------- |
| Cross-agent automation update                           | Elevation of privilege | A token scoped for `write:automation` on agent B uses `record_automation_run` against agent A's known external_id. A naïve writer would let the update land.          | `vault.automations.record_run` filters by `agent_id` at the SQL layer and raises `UnknownAutomationError` on miss; the MCP boundary surfaces this as `ValidationError` so the caller sees an explicit denial. The MCP layer also calls `_enforce_same_agent_profile_ref` on `register_automation` so a write-scoped caller cannot plant an automation that points at another agent's profile. Tests `test_record_automation_run_blocks_cross_agent_update` and `test_register_automation_blocks_cross_agent_ref` pin both boundaries. | v0.5-P5     |
| Storage-only boundary erosion (scheduler in the daemon) | Trust posture          | A future "convenience" feature adds an in-process timer or outbound trigger when an automation's `cadence` elapses, turning Tessera into a scheduler runtime.         | ADR 0020 §Boundary statement: Tessera registers automations as data; runners execute them. The non-goals canon (`docs/non-goals.md`) explicitly bars scheduler runtimes and outbound calls except configured adapters. A future surface that wants in-process scheduling opens its own ADR before extending the boundary. The CI gate `no-telemetry-grep` already rejects HTTP client imports outside the allowlisted adapter / CLI paths so an outbound trigger cannot land silently.                                                                                                                                                                                                | v0.5-P5     |
| Bypass write-path validation via raw `capture`          | Trust posture          | A write-scoped caller targets `facet_type='automation'` through the generic `capture` tool, sidestepping the closed `{agent_ref, trigger_spec, cadence, runner}` contract. | The MCP `capture` tool rejects `facet_type='automation'` with an explicit `register_automation` redirect (mirror of the V0.5-P2 `agent_profile` gate). Tests `test_capture_blocks_raw_automation_writes` pin the gate. Routing every automation write through `register_automation` upholds the "every stored row is parseable" invariant the storage-only registry depends on — a malformed row would later break `_row_to_automation` for every read.                                                                                                                                                                                                                          | v0.5-P5     |
| Free-form `last_result` leakage into audit chain        | Information disclosure | A runner records `last_result="customer X disputed invoice 12345 at https://internal.example/dispute/abc"` after a run. The string contains user content the chain should not carry. | `automation_run_recorded` payload allowlist is locked to `{result_bucket, last_run_at}`. The runner-supplied `last_result` is bucketed into `success` / `partial` / `failure` / `other` for the audit row; the full string lands on the row's `metadata` column where the standard §S4 protections (encrypted vault, soft-delete, no telemetry) apply, but the chain itself never carries free-form caller prose. Test `test_record_run_buckets_free_form_result_as_other` pins the contract.                                                                                                                                                                                                                                                              | v0.5-P5     |

### S4 — Audit log (A7)

| Threat                                                                          | Category               | Vector                                                                                                                                                                  | Mitigation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | Status                  |
| ------------------------------------------------------------------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| Accidental corruption / non-recomputed tampering of audit_log                   | Tampering / repudiation | An attacker with vault write access deletes, modifies, reorders, or inserts rows but does not recompute the chain hash; or a buggy migration / partial restore corrupts the table. | V0.5-P8 forward linear hash chain per ADR 0021. Each row carries `prev_hash` (the prior row's `row_hash`, or `''` at genesis) and `row_hash = sha256(prev_hash` \|\| `canonical_json(event))`. The single canonical insert path (`tessera.vault.audit_chain.audit_log_append`) reads the head, computes the new hash, and inserts inside one transaction; CI gate `audit-chain-single-writer` rejects any direct `INSERT INTO audit_log` outside that function. `tessera audit verify` walks the chain end-to-end; deletion / modification / reorder / insertion all break the walk. **Boundary**: this **does not** detect tampering by an attacker who can recompute hashes (the chain payload is unkeyed, the canonicalizer is published), pre-upgrade tampering, or loss of the entire audit_log table. Public language stays "tamper-evidence within the stated claim boundary," never "tamper-proof."                                                  | v0.5 (P8)               |
| Recompute-capable attacker rewrites audit_log                                   | Tampering              | Attacker with passphrase + write access to the vault recomputes every `row_hash` from the published canonicalizer.                                                       | Out of scope at v0.5 by explicit claim boundary (ADR 0021 §Security claim — exact boundary). Mitigations require keyed/signed/anchored variants (HMAC chain with key outside vault, signed checkpoints, external anchors); all three are reviewed-but-not-built designs at v0.5. A v1.0 follow-up ADR commits one before promoting the trust language.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | v1.0 (deferred)         |
| Pre-upgrade tampering                                                           | Tampering              | An attacker edited audit_log rows on a vault that predated V0.5-P8; the migration backfill chains those rows in their stored order at upgrade.                          | Documented as a known limitation in `docs/migration-contract.md` and in the `tessera audit verify` help text. The genesis row id reported by `verify` flags which rows predate the chain; rows written after the upgrade are fully chained. No retroactive fix exists — the operations were recorded, not replayed. Real-world mitigation: take a backup before the v3 → v4 migration so the pre-upgrade snapshot is verifiable against the post-upgrade chain.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | v0.5 (boundary)         |
| Loss of the entire audit_log table                                              | Repudiation            | Attacker drops the table entirely; `verify` reports zero rows but the chain has no external anchor to detect the loss.                                                 | Out of scope at v0.5 by ADR 0021. Detection requires an external anchor (signed checkpoints, off-host backup, exported chain heads). v1.0 may add this as opt-in. Real-world mitigation: BYO sync (V0.5-P9) carries the chain across hosts; periodic offline backups give the user an independent reference.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                | v1.0 (deferred)         |
| Audit log exhausts disk                                                         | Denial of service      | Runaway agent writes audit entries without bound                                                                                                                         | Rate-limit audit log writes per capability token; rotate to a cold archive table at configurable row count.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  | v0.3 (deferred)         |
| Audit log PII leak                                                              | Information disclosure | Log payloads contain secret content hashed into facets                                                                                                                   | Payloads store IDs and operation metadata, not facet content. Explicit allow-list of keys in the payload JSON. The chain's `canonical_json` of every event is bound by the same allowlist — there is no path to the chain that bypasses §S4's content-exclusion contract.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  | v0.1                    |
| `compiled_artifact_marked_stale` payload leakage                                | Information disclosure | A future caller passes free-text justification ("staleness reason: user said the procedure was wrong") into the cascade audit row, leaking source content into the chain. | The op's allowlist locks payload keys to `{source_external_id, source_op}` — both internal vault identifiers, never user content. `source_op` itself is restricted to `{facet_inserted, facet_soft_deleted, skill_procedure_updated}` at the staleness primitive level so a buggy caller cannot smuggle a free-text label through the op field. Adding a fourth `source_op` value requires updating both the staleness primitive's allowlist and the `_PAYLOAD_ALLOWLIST` declaration in the same commit, by §S4 protocol.                                                                                                                                                                                                                                                                                                                                                                                                                                  | v0.5-P6                 |
| `automation_run_recorded` payload leakage                                       | Information disclosure | A runner supplies free-form `last_result` ("customer X disputed invoice 12345") to `record_automation_run`. A naïve audit emitter would land that string in the chain. | The op's allowlist locks payload keys to `{result_bucket, last_run_at}`. `result_bucket` is bucketed at the storage layer to `{success, partial, failure, other}` — free-form values become `"other"` and the original prose stays on the row's `metadata` column (encrypted with the rest of the vault). `last_run_at` is the ISO-8601 timestamp the runner supplied, validated against a fixed pattern. Test `test_record_run_buckets_free_form_result_as_other` pins the bucketing; the §S4 boundary is preserved.                                                                                                                                                                                                                                                                                                                                                                                                                                          | v0.5-P5                 |

## Out of scope

- **Nation-state adversaries with physical access and coercion.** Tessera does not claim duress protection.
- **Kernel or hypervisor compromise.** If the OS is compromised, the vault is compromised. No mitigations at user-space can change this.
- **Side-channel attacks on the encryption key while unlocked.** Mitigations exist (mlock, no swap) but a rootkit on an unlocked machine wins.
- **Social engineering of the user's passphrase.** The passphrase is user-managed.
- **Deepfake of user voice instructing the agent.** Tessera stores identity; it does not verify the identity of the humans instructing the agent.

## Mitigation verification

Each mitigation above has a corresponding test or explicit verification artifact. The mapping is tracked in `tests/security/README.md` (to be created in v0.1 dev cycle). A mitigation without a test does not count as delivered.

| Verification type           | Examples                                                                                 |
| --------------------------- | ---------------------------------------------------------------------------------------- |
| Unit / integration test     | Token hashing, scope enforcement, FK constraints, CSRF header check, hard-delete cascade |
| Network policy test in CI   | Outbound block test, localhost-only binding                                              |
| Manual audit on release     | Keyring integration, Origin-header rejection of browsers, mlock of unlock key            |
| External review (post-v0.3) | Encryption primitive choice, HMAC chain construction                                     |

## Revision policy

- Threats added when a new feature is proposed; the PR that adds the feature updates this document.
- Threats removed only by explicit deprecation of the surface (e.g., if HTTP MCP is replaced by Unix-socket-only, S3 threats are reworked).
- Mitigation deadlines slip with a dated note; slipping past a major version requires an explicit reassessment.
